Test targets: Nginx (Reverse Proxy) → Flask Application (Backend)

The goal is to start from a canonical blocked URL (401/403) and verify whether an alternate variant returns 200 OK and exposes restricted content.

1) Admin area — off-by-slash bypass (403)

  • Canonical endpoint: http://fbps.com/admin/ (expected: 403)
  • Bypass candidate: http://fbps.com/admin (expected: 200)

Bypass Reason

Because the deny rule matches only the /admin/ prefix, the /admin variant does not hit it and falls back to location /, which proxies the request to the backend. If the backend accepts /admin and /admin/ as equivalent, /admin returns 200 OK while /admin/ is blocked.

Vulnerable Nginx Configuration:

location / {
    proxy_pass http://app_upstream;
}


location /admin/ {
    return 403;
}

2) Private area — authentication bypass (401)

  • Canonical endpoint: http://fbps.com/private/ (expected: 401)
  • Bypass candidate: http://fbps.com/private/index.html (expected: 200)

Bypass Reason:

Basic Auth is enforced only on the exact-match location /private/; requests to /private/index.html do not hit that rule and fall back to a more permissive proxy location, reaching the backend without authentication.

Vulnerable Nginx Configuration:

location / {
    proxy_pass http://app_upstream;
}


location = /private/ {
    auth_basic "Restricted";
    auth_basic_user_file /etc/nginx/.htpasswd;
    proxy_pass http://app_upstream;
}

3) Internal File — regex vs prefix mismatch (403 → 200)

  • Canonical endpoint: http://fbps.com/static/internal/users (expected: 403)
  • Bypass candidate: http://fbps.com/static/internal/users.json (expected: 200)

Bypass Reason:

The prefix deny rule location /static/ is overridden by the case-insensitive regex location ~* \.json$; as a result, requests ending in .json are routed to the backend even under /static/, returning 200 OK instead of 403.

Vulnerable Nginx Configuration:

location /static/ {
    return 403;
}

location ~* \.json$ {
    proxy_pass http://app_upstream;
}

4) Forwarded headers — X-Forwarded-For

  • Canonical endpoint: http://fbps.com/local/
  • Bypass candidate: curl -s http://fbps.com/local/ -H "X-Forwarded-For: 127.0.0.1"

Bypass Reason:

Access control relies on $http_x_forwarded_for, which is derived from a client-controlled header. By sending X-Forwarded-For: 127.0.0.1, the request satisfies the check and is proxied to the backend, resulting in 200 OK.

Vulnerable Nginx Configuration

location /local {
    if ($http_x_forwarded_for !~ "^127\.0\.0\.1(,|$)") {
      return 403;
    }

    proxy_pass http://app_upstream;
}

5) Trim Inconsistencies

  • Canonical endpoint: http://fbps.com/trim/
  • Bypass candidate: curl http://fbps.com --request-target "$(printf '/trim\xa0')"

Bypass Reason:

The reverse proxy applies the deny rule to an “exact” request path, but the backend receives a differently normalized/trimmed path. By using a non-standard trailing character in the request-target (e.g., a non-breaking space), the proxy and the backend may interpret the path differently, allowing the request to bypass the intended 403 check.

Vulnerable Nginx Configuration

location = /trim {
    return 403;
}

location = /trim/ {
    return 403;
}

6) API version downgrade

  • Canonical endpoint: http://fbps.com/api/v2/secrets
  • Bypass candidate: http://fbps.com/api/v1/secrets

Bypass Reason:

Access control is enforced only on the latest API version under /api/v2/, while legacy endpoints such as /api/v1/secrets remain reachable and return 200 OK. By “downgrading” the request to an older version, an attacker can access the same sensitive data without hitting the restrictive rule.

Vulnerable Nginx Configuration

location / {
    proxy_pass http://app_upstream;
}

location /api/v2/ {
    return 403;
}

7) Case mismatch

  • Canonical endpoint: http://fbps.com/services/myservice/status
  • Bypass candidate: http://fbps.com/SERVICES/myservice/status

Bypass Reason:

The reverse proxy enforces access control only on the lowercase path /services/myservice, because its matching rules are case-sensitive. The backend, however, accepts this service path in a case-insensitive manner. As a result, any differently cased variant (e.g. /SERVICES/MYSERVICE/STATUS) does not match the proxy’s restriction but still resolves to the same backend endpoint, allowing the request to succeed with 200 OK.

Vulnerable Nginx Configuration

location / {
    proxy_pass http://app_upstream;
}

location /services/myservice {
    return 403;
}

8) User-Agent allowlist

  • Canonical endpoint: http://fbps.com/user_agent
  • Bypass candidate: curl -i -H "User-Agent: FBps/1.0" http://fbps.com/user_agent

Bypass Reason:

Access to /user_agent is restricted by the reverse proxy unless the request’s User-Agent matches an allowlist pattern. By setting the header to an allowed value such as FBps/1.0, a client satisfies the regex-based rule and reaches the backend, receiving 200 OK.

Vulnerable Nginx Configuration

map $http_user_agent $ua_allowed {
  default 0;
  ~^FBps(\b|/|$) 1;
}

location = /user_agent {
    if ($ua_allowed = 0) { 
        return 403; 
    }
    
    proxy_pass http://app_upstream;
}
